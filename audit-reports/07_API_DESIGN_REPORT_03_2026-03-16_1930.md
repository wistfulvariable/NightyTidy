## 1. Executive Summary

NightyTidy exposes **20 HTTP/WS endpoints** across five server implementations:

| Server | File | Endpoints | Purpose |
|--------|------|-----------|---------|
| GUI Server | `gui/server.js` | 11 POST + 1 GET (static) | Desktop GUI backend |
| Dashboard | `src/dashboard.js` | 2 GET + 1 POST | CLI progress display |
| Standalone Dashboard | `src/dashboard-standalone.js` | 2 GET + 1 POST | Orchestrator mode dashboard |
| Firebase Cloud Functions | `functions/src/*.ts` | 1 POST + 2 GET | Remote API (webhook, runs, health) |
| Local Agent WebSocket | `src/lib/websocket.ts` (client) | WS + 1 GET discovery | Real-time agent communication |

### Overall Assessment

The **local server APIs** (GUI, Dashboard, Standalone) demonstrate **excellent internal consistency** — uniform naming (`/api/kebab-case`), uniform response format (`{ ok, error }`), security headers everywhere, CSRF where needed, and body size limits.

The **Cloud Functions API** follows a different but internally consistent convention — Firebase-idiomatic, with JSON responses, Bearer token auth, and rate limiting.

The **cross-layer inconsistency** between local and cloud APIs is **intentional and appropriate** — they serve different consumers (localhost GUI vs. remote clients) with different security models.

### Key Findings

1. **Cloud Functions error responses are plain text, not JSON** — `res.status(401).send('Unauthorized')` vs local API's `{ ok: false, error: "..." }`. Not a bug, but an inconsistency.
2. **SSE `/events` endpoints lack security headers** on both Dashboard and Standalone Dashboard.
3. **`/runs` Cloud Function has no rate limiting** — unlike `webhookIngest` which has 60/min per user.
4. **`status` Cloud Function accepts any HTTP method** — no method guard unlike the other two functions.
5. **Cloud Functions `webhookIngest` does minimal request body validation** — accepts any shape as long as `body.run.id` exists.
6. **`/runs` has pagination ceiling (100) but no consumer-controllable page size** — fixed at hardcoded limit.

**Assessment**: The GUI server's "POST for everything" convention is documented and intentional. It prioritizes simplicity and security (no sensitive data in URLs) over REST semantics. The Cloud Functions follow standard REST conventions. No issues.

### 4.2 Status Code Audit

|--------|-------|----------|

3. **`/runs` Cloud Function returns 200 with empty array for no results** — ✅ Correct (not 404).

## 5. Error Response Consistency

### 5.1 Error Response Formats

**Local Servers (GUI, Dashboard, Standalone)**:

```json
// Standard error (JSON)
{ "ok": false, "error": "Human-readable message" }

// Or JSON with error only (Dashboard stop):
{ "error": "Invalid token" }
```

**Cloud Functions**:

```
// Auth errors — plain text
401: "Unauthorized"
401: "Invalid token"
405: "Method not allowed"
429: "Too many requests"

// Success — JSON
200: { "ok": true }
200: { "runs": [...] }
```

### 5.2 Inconsistencies

| Location | Current Format | Target Format | Breaking Change? |
|----------|---------------|---------------|-----------------|
| Cloud: 401 response | `res.send('Unauthorized')` (plain text) | `{ ok: false, error: "Unauthorized" }` (JSON) | Yes — consumer checks response format |
| Cloud: 405 response | `res.send('Method not allowed')` (plain text) | `{ ok: false, error: "Method not allowed" }` (JSON) | Yes |
| Cloud: 429 response | `res.send('Too many requests')` (plain text) | `{ ok: false, error: "Too many requests" }` (JSON) | Yes |
| Dashboard stop: 403 | `{ error: "Invalid token" }` (no `ok` field) | `{ ok: false, error: "Invalid token" }` | Minor |
| Standalone stop: 200 | `{ ok: true, message: "Stop not supported..." }` | `{ ok: true }` (no extra message) | No |

### 5.3 Error Quality Assessment

- **Messages are specific and helpful**: "No command provided", "Not an allowed file", "No process id provided"
- **No sensitive info leaked**: No stack traces, SQL errors, or internal paths in error responses
- **Fail-on-first validation**: Each endpoint returns one error at a time (acceptable for this API size)
- **No machine-readable error codes**: Errors use only human-readable strings (acceptable for internal APIs)

### 5.4 Dashboard Stop Error Format Deviation

Both `dashboard.js` and `dashboard-standalone.js` return `{ error: "Invalid token" }` without the `ok` field for CSRF failures. The GUI server always includes `ok`. This is a minor inconsistency — the dashboard stop is consumed only by the embedded HTML, not by external clients.

## 6. Pagination

### 6.1 List Endpoints

| Endpoint | Paginated? | Strategy | Params | Default | Max | Metadata? |
|----------|-----------|----------|--------|---------|-----|-----------|
| Cloud `/runs` | Partial | Firestore limit | None | 100 | 100 | None |
| Client `getRuns()` | Partial | Firestore limit | `maxResults` | 20 | None enforced | None |
| Client `getPresets()` | No | Full fetch | None | — | — | None |
| Client `getWebhooks()` | No | Full fetch | None | — | — | None |
| Client `getGlobalStepStats()` | No | Full fetch | None | — | — | None |

### 6.2 Assessment

- **`/runs` Cloud Function**: Hardcoded limit of 100, ordered by `startedAt` DESC. No cursor pagination, no consumer-controllable page size, no pagination metadata (total count, hasMore). For the expected data volume (dozens to hundreds of runs per user), this is adequate.

- **Client-side `getRuns()`**: Uses a different default (20) than the Cloud Function (100). The client function's `maxResults` parameter is a Firestore-level limit, not an API-level parameter. This works because the client reads Firestore directly, not via the Cloud Function.

- **Unbounded collections**: `getPresets()`, `getWebhooks()`, `getGlobalStepStats()` fetch all documents without limits. Given expected cardinality (< 50 presets, < 20 webhooks, < 100 step stats), this is acceptable.

### 6.3 Recommendation

No immediate action needed. If run history grows large, consider adding cursor-based pagination to `/runs` with `startAfter` support and pagination metadata. This is a future enhancement, not a current bug.

### 7.1 Validation Patterns

| Layer | Library | Location | Consistent? |

| GUI Server | None (inline) | Start of handler | ✅ |
| Cloud Functions | None (inline) | Start of handler | ✅ |

No validation library (zod, joi, yup) is used anywhere. All validation is inline at handler start with early returns. This is consistent and appropriate for the API size.

### 7.2 Validation Coverage

**GUI Server** — Well covered:

| Endpoint | Fields Validated | Gap? |
|----------|-----------------|------|
| `/api/run-command` | `command` required | No |
| `/api/kill-process` | `id` required | No |
| `/api/read-file` | `path` required | No |
| `/api/delete-file` | `path` required + allowlist | No |
| `/api/log-error` | `message` required | No |
| `/api/config` | No input | N/A |
| `/api/select-folder` | No input | N/A |
| `/api/heartbeat` | No input | N/A |
| `/api/log-path` | No input | N/A |
| `/api/exit` | No input | N/A |

**Cloud Functions** — Partial:

| Endpoint | Fields Validated | Gap? |
|----------|-----------------|------|
| `/webhookIngest` | Method (POST), Auth token | ⚠️ Body not validated beyond `body.run?.id` |
| `/runs` | Method (GET), Auth token, `projectId` type check | No |
| `/status` | None | ⚠️ No method guard |

### 7.3 Validation Gaps

1. **`webhookIngest` body validation**: The endpoint accepts any request body shape. If `body.run?.id` is falsy, the entire body is silently ignored (200 returned). The `event` field is not validated against known values — unknown events fall to the generic handler. The `step.number`, `step.cost`, `step.duration` fields are type-checked inline (`typeof === 'number'`) for the global stats, but not for the step document write.

   **Risk**: Low. The endpoint is behind Firebase Auth, so only authenticated users can call it. Malformed payloads from the NightyTidy CLI agent would indicate a bug in the agent, not a security issue.

2. **`status` method guard**: The `/status` endpoint responds to any HTTP method. Adding `if (req.method !== 'GET') { res.status(405).send('Method not allowed'); return; }` would align it with the other Cloud Functions.

   **Risk**: Very low. Health checks are frequently probed by monitoring tools with various methods.

### 7.4 Error Format Consistency

All validation failures return consistent formats within each layer:
- GUI: `{ ok: false, error: "<message>" }` with 400 status
- Cloud: Plain text with 401/405/429 status
- Dashboard: JSON with 403/413 status

## 8. Miscellaneous API Quality

### 8.1 Rate Limiting

| Endpoint | Rate Limited? | Limit | Headers? |
|----------|--------------|-------|----------|
| `/webhookIngest` | ✅ | 60/min/user | No `X-RateLimit-*` headers |
| `/runs` | ❌ | — | — |
| `/status` | ❌ | — | — |
| GUI endpoints | ❌ (localhost) | — | N/A |
| Dashboard endpoints | ❌ (localhost) | — | N/A |

**Assessment**:
- `webhookIngest` rate limiting is Firestore-backed with a sliding window. Implementation is correct. Missing `X-RateLimit-Remaining` / `X-RateLimit-Reset` headers — consumers (the CLI agent) don't use them, so this is low priority.
- `/runs` has no rate limiting. Since it's a read-only Firestore query behind auth, Firebase's built-in quotas provide implicit protection.
- Local endpoints don't need rate limiting (localhost only, single user).
- Rate limiter fails open (`catch { // fail open }`) — this is intentional and documented.

### 8.2 Versioning

No API versioning exists. This is acceptable because:
- Local APIs: Internal to the same application (GUI ↔ server)
- Cloud Functions: Internal consumer (web dashboard) deployed simultaneously
- WebSocket: Internal to the same application
- If external consumers are ever added, versioning should be introduced.

### 8.3 Content Types

| Layer | Sets `Content-Type`? | JSON responses? |
|-------|---------------------|-----------------|
| GUI Server | ✅ `application/json; charset=utf-8` | ✅ via `sendJson()` helper |
| Dashboard | ✅ `application/json` (stop), `text/html` (/), `text/event-stream` (/events) | ✅ |
| Standalone | ✅ Same as Dashboard | ✅ |
| Cloud Functions | ✅ Implicit via `res.json()` | ✅ (for JSON responses) |

**Issue**: Cloud Function error responses use `res.send(string)` which sets `Content-Type: text/html` by Express default, while success responses use `res.json()` which sets `Content-Type: application/json`. This means 401/405/429 responses have a different content type than 200 responses.

### 8.4 Idempotency

| Endpoint | Idempotent? | Notes |
|----------|-------------|-------|
| `webhookIngest` `run_started` | ✅ | Uses `runRef.set()` (full replace) |
| `webhookIngest` `step_completed` | ⚠️ | Uses `FieldValue.increment()` — replaying causes double-counting |
| `webhookIngest` `run_completed` | ✅ | Uses `runRef.set({ merge: true })` — same result on replay |
| GUI `/api/kill-process` | ✅ | Returns `ok: true` even if process already dead |
| GUI `/api/delete-file` | ✅ | Returns `ok: true` even if file already gone |
| Dashboard `/stop` | ✅ | `try { onStop() } catch {}` handles double-stop |

**Issue**: `step_completed` events use `FieldValue.increment()` for `completedSteps`, `totalCost`, `duration`, and `filesChanged`. Replaying the same event (e.g., network retry) would increment these values again. This is a known trade-off — at-least-once delivery with eventual consistency. For the NightyTidy use case (single CLI agent per run), duplicates are extremely unlikely.

### 8.5 Discoverability

- No API index endpoint or HATEOAS links (appropriate for internal APIs)
- Local agent has `/auth-info` discovery endpoint for port/token negotiation
- API Design Guide exists at `docs/API_DESIGN_GUIDE.md`
- API contract tests exist at `functions/src/__tests__/apiContracts.test.ts`
