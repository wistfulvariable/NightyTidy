## 1. Executive Summary

NightyTidy exposes **17 HTTP endpoints** across three server implementations:
- **GUI Server** (`gui/server.js`) — 11 API endpoints for desktop application
- **Dashboard Server** (`src/dashboard.js`) — 3 endpoints for CLI progress display
- **Standalone Dashboard** (`src/dashboard-standalone.js`) — 3 endpoints for orchestrator mode

The API surface demonstrates **excellent consistency**:
- ✅ **Uniform naming convention**: All endpoints use `kebab-case` URL paths
- ✅ **Consistent HTTP methods**: POST for mutations, GET for reads, SSE for streaming
- ✅ **Uniform error response format**: `{ ok: false, error: "..." }` everywhere
- ✅ **Complete security headers**: All responses include CSP, X-Frame-Options, X-Content-Type-Options
- ✅ **CSRF protection**: Dashboard stop endpoints require token validation
- ✅ **Consistent body limits**: 1 MB (GUI), 1 KB (dashboard stop)

### Minor Issues Found

1. **SSE endpoints missing security headers** — `/events` uses `text/event-stream` without full security header set
2. **Inconsistent error status codes** — Some endpoints return 400 for missing params, others return 200 with `ok: false`
3. **No API versioning** — All endpoints are unversioned (acceptable for internal-only API)

## 4. HTTP Method & Status Code Correctness

### 4.1 HTTP Method Audit

| Method | Endpoint | Semantic Correctness |
|--------|----------|---------------------|
| GET | `/` | ✅ Read-only, serves static HTML |
| GET | `/<path>` | ✅ Read-only, serves static files |
| GET | `/events` | ✅ SSE stream (read-only, no side effects) |
| POST | `/api/config` | ⚠️ Could be GET (read-only, no body needed) |
| POST | `/api/select-folder` | ✅ Triggers side effect (OS dialog) |
| POST | `/api/run-command` | ✅ Creates process, triggers execution |
| POST | `/api/kill-process` | ✅ Mutation (kills process) |
| POST | `/api/read-file` | ⚠️ Could be GET with path param, but body allows complex paths |
| POST | `/api/delete-file` | ✅ Mutation (deletes file) |
| POST | `/api/heartbeat` | ✅ Mutation (updates server state) |
| POST | `/api/log-error` | ✅ Mutation (writes log) |
| POST | `/api/log-path` | ⚠️ Could be GET (read-only) |
| POST | `/api/exit` | ✅ Mutation (shuts down server) |
| POST | `/stop` | ✅ Mutation (stops run) |

**Notes**:
- Using POST for read-only endpoints (`/api/config`, `/api/read-file`, `/api/log-path`) is **intentional design**: POST allows body parameters without URL encoding issues, especially for file paths containing special characters. This is a reasonable tradeoff.
